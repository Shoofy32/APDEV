<b> MCO Project Forum Specifications: </b>
<details>
<summary>Specifications</summary>
<ul>
    <li> View all posts  </li>
      <ul>
        <li> Upon visiting the web page, an unregistered visitor may see the 15-20 most 
recently* uploaded post titles and a post description snippet. The user can 
see the next set of uploaded posts, it is up to the group whether to 
implement this in the same (auto-load) or another page. Clicking the post 
title will let the user view the post and the comments (see: view a post).   </li>
        <li> * The user may also see the most popular post based on a calculated 
ranking.  </li>
      </ul>

  <li> Register </li>
      <ul>
        <li> A visitor must register if they want to post or comment. Here, a visitor 
must enter their username and their password.    </li>
      </ul>

  <li> View a user profile </li>
      <ul>
        <li> Each user has their own page which shows their profile publicly. On the 
same page, a visitor may see the user’s username, profile picture, and a 
short description. They may also see a portion of the user’s latest posts 
and comments. The visitor may opt to see the rest of the posts and 
comments of the user.  </li>
      </ul>

  <li> Edit Profile  </li>
      <ul>
        <li> A user that is logged in may edit their user profile, wherein they can add 
/ modify a profile picture, and provide a short description (can be left 
empty).  </li>
      </ul>

  <li> Login </li>
      <ul>
        <li> After registering properly, a visitor may log-in. Upon logging in, the user 
can start posting and commenting. The user is given the option to be 
“remembered” by the website. When the user chooses this option, every 
log in and visit to the website will extend their “remember” period by 3 
weeks. </li>
      </ul>

  <li> Logout </li>
      <ul>
        <li> The user may log out from their account. This should cut short the 
“remember” period if it exists, and clears any session-related data.   </li>
      </ul>

  <li> Post </li>
      <ul>
        <li> A user may make a text post. They must give a title for the post, and the 
body of the post. Additional points will be given for allowing markup (e.g., 
rich text editing) without the risk of cross site-scripting.  </li>
      </ul>

  <li> View a post </li>
      <ul>
        <li> A user may view any post they have a link to. This will load the title, the 
body of the post, and the comments of the post.   </li>
      </ul>

  <li> Comment </li>
      <ul>
        <li> A user can comment on any post, including their own. They may also reply 
to another user’s comments, and the comments can nest indefinitely.  </li>
      </ul>

  <li> Edit / Delete a post  </li>
      <ul>
        <li> The owner of the post may edit their posts at any point. Edited posts 
should have an indication that it has been edited.   </li>
        <li> The owner of the post may delete their post as well.    </li>
      </ul>

  <li> Edit / Delete a comment </li>
      <ul>
        <li> The owner of the comment may edit their comments at any point. Edited 
comments should have an indication that it has been edited.   </li>
        <li> The owner of the comment may delete their comment.  </li>
      </ul>

  <li> Upvote / Downvote </li>
      <ul>
        <li> Each user has their own page which shows their profile publicly. On the 
same page, a visitor may see the user’s username, profile picture, and a 
short description. They may also see a portion of the user’s latest posts 
and comments. The visitor may opt to see the rest of the posts and 
comments of the user.  </li>
      </ul>

  <li> Search </li>
      <ul>
        <li> A visitor/user can search for posts by similarities in the title or the post 
body. By entering a search phrase/word, all posts containing it will appear 
as results.   </li>
      </ul>

  <li> General </li>
      <ul>
        <li> Good user experience. Visitors can easily navigate without help, all 
information is easy to access. Good visual design. Design suits the theme 
of the application, and is cohesive and consistent across the whole 
application.  </li>
      </ul>
</ul>
</details>

<b> Your task is to create the front-end portion of your selected web application. </b>

<b> Filename format: </b>  CCAPDEV-Phase1-Group<#>.zip 

<ul>
  <li>For this phase, the group is required to develop the front-end view of their 
selected web application using HTML, CSS, and JavaScript.) </li>
  <li>Aside from base CSS and JavaScript, the group may also use interface libraries 
like Bootstrap, meteor, and React, among others.  </li>
  <li>All views by all users should be visible and navigable from the index webpage. 
The group is required to implement the front-end view of all features described 
in the project specifications document. The back-end logic of the minimum 
features is not yet required to be implemented in this phase. </li>
  <li>Data may be hardcoded for now. There should be at least 5 sample data for each 
applicable feature. Data encoded should be life-like, thus no lorem ipsum 
generated paragraphs. For example, in a forum web application, there should be 
at least 5 sample users,  5 sample posts, and 5 sample comments across the web 
application. </li>
  <li>Students should collaborate through their group’s GitHub repository. The 
repository will be used as reference in case problems arise with group 
collaborations. </li>
</ul>
